# Testing Patterns

- Test files live **alongside** what they test: `students-import.helpers.spec.ts` next to `students-import.helpers.ts`, `heatmap.service.spec.ts` next to `heatmap.service.ts`. Jest's `testRegex` (`apps/api/package.json`) matches any `*.spec.ts` under `src/`.
- Run the suite from `apps/api`: `pnpm test` (→ `jest`). To run one file, pass its name/path to Jest directly — e.g. `pnpm test students-import.helpers` or `pnpm test -- src/heatmap/heatmap.service.spec.ts`. There is no `test:file` script.
- `setupFiles` runs `apps/api/src/test-setup.ts` before every suite — it only loads the monorepo root `.env` via `dotenv`. There is no shared DB-seeding/reset setup.
- **There is no live test database wired up in `apps/api` today.** Every existing `.spec.ts` does one of:
  - tests pure logic with no DB involved at all — `students/students-import.helpers.spec.ts` calls `normalizeRut`, `parseCursoLabel`, `parseStudentRosterCsv` directly, no service, no mocks; or
  - hand-builds a minimal fake `Database` and injects it into the service constructor — `heatmap/heatmap.service.spec.ts` defines a `makeDb(selectResults: unknown[][])` that fakes the `.select().from().where()...` chain and resolves canned rows per call in call order; `items/items.service.spec.ts` passes `{} as never` as `db` for tests that only exercise DB-free methods like `assertVisible`/`assertEditable`.

  CLAUDE.md §10.2 ("no mockear la base de datos en tests de integración — usar una DB de test real con datos de seed") states the target for integration tests; it is **not** what's implemented in `apps/api` yet — treat it as the direction to move in, not a mechanism you can invoke today. Follow the existing pattern (fake the `Database` shape per test, or test pure helpers with zero DB) rather than assuming a seeded test Postgres exists.
- F1 has no queue/SQS — async work is tracked in the `import_jobs` table with polling, not BullMQ (CLAUDE.md §12), so there's nothing queue-related to mock.
- `jest.mock` shows up rarely, and only to stub one specific export, not a whole subsystem: `assistant/assistant.service.spec.ts` and `auth/auth.service.switch-org.spec.ts` do `jest.mock('@soe/db', () => ({ ...jest.requireActual('@soe/db'), /* override one export */ }))`; `curriculum-retriever/structured-curriculum-retriever.spec.ts` mocks `drizzle-orm`'s predicate builders (`eq`, `ne`, ...) so it can assert on the query conditions a service builds. There's no established pattern yet for mocking LLM calls (Anthropic/Gemini) or S3 — if you need one, mock narrowly at this same per-export level, don't introduce a global mock module.
- For exercising a controller end-to-end, use `supertest` — see `assistant/assistant.controller.spec.ts` and `privacy/privacy.controller.spec.ts`.
